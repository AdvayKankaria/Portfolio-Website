════════════════════════════════════════════
3D SCENE (React Three Fiber)
════════════════════════════════════════════

Load the entire Three.js scene with next/dynamic and ssr: false. It must never block the main thread.

Scene composition:
  3–5 floating geometric primitives: IcosahedronGeometry, TorusKnotGeometry, OctahedronGeometry.
  Each rotates slowly on its own axis (different speeds, different axes).
  Mouse parallax: on mousemove, all geometries shift position by ±5% of their base position. Lerp factor: 0.05 (smooth, not instant).
  Material: MeshStandardMaterial with metalness 0.8, roughness 0.2, emissive color from MASTER.md accent.
  Lighting: single AmbientLight (intensity 0.4) + single PointLight (intensity 1.2, position [10, 10, 10]).

Particles:
  BufferGeometry with 800–1200 points. Never per-instance geometry.
  Slow drift upward, wrap at top edge.
  Color: MASTER.md accent at 40% opacity.

Performance rules (non-negotiable):
  No postprocessing effects (Bloom, SSAO, etc.) unless devicePixelRatio === 1 and FPS > 58.
  Set pixelRatio to Math.min(window.devicePixelRatio, 2).
  Dispose all geometries and materials on component unmount.
  Use useFrame delta for time-based animation — never Date.now().
  Target 60fps on a mid-range laptop (test with DevTools CPU throttle 4x).

Fallback: if WebGL context creation fails, render an animated CSS gradient background using the MASTER.md palette. No error visible to user.
Reduced motion: render scene static (no animation loop) — just the geometries in their start positions.

─────────────────────────────────────────
WHAT THIS THEME IS
─────────────────────────────────────────

Graphite Mono + Red = precision engineering meets controlled intensity.

Graphite: cool steel, machined aluminum, carbon fiber.
Not warm gray. Not neutral gray. Metallic gray with subtle cool blue
undertones (hue ~260) that read as precision-engineered, not muted.

Mono: pure black and white editorial contrast pushed to its limit.
Every element is graphic. Typography becomes design. Nothing hides.

Red: the single chromatic signal in an achromatic system.
Not a decoration — a scalpel. Used only where it must demand attention:
primary CTA buttons, active states, hover reveals, the cursor ring.
The contrast between near-black surfaces and pure red is the visual
identity of this portfolio. Think: dark terminal with a red cursor.
It should feel like a warning light on precision equipment — purposeful,
not alarming.

Together: a dark engineering portfolio with one precise strike of red.
The kind of site that makes a recruiter pause and look twice.

     Active: background shifts to var(--red-600), 100ms ease
     Border: none — the fill is the statement
     Glow on hover: box-shadow: 0 0 30px -8px oklch(0.58 0.22 27 / 0.5)
     Font: font-mono, letter-spacing 0.05em, text-transform uppercase
     Example: "VIEW WORK", "CONTACT ME", "DOWNLOAD CV"

  2. GHOST BUTTONS (secondary CTAs)
     Background: transparent
     Border: 1px solid var(--red-500)
     Text: var(--red-400)
     Hover: background fills to oklch(0.58 0.22 27 / 0.08)
     The red border on near-black background is the signature look.

  3. ACTIVE NAVIGATION INDICATOR
     The underline or left-border on the currently active nav link.
     Not the text — just the indicator line.
     Color: var(--red-500)

  4. CUSTOM CURSOR RING
     The outer ring of the custom cursor: stroke var(--red-500)
     On hover over interactive elements: ring expands, fills to
     oklch(0.58 0.22 27 / 0.15)

  5. SCROLL PROGRESS BAR
     The thin top progress line: var(--red-500)

  6. HOVER REVEAL ON PROJECT CARDS
     When hovering a project card, a thin 1px red left-border appears
     (border-left: 2px solid var(--red-500)) and the card title shifts
     to var(--red-400). The rest of the card stays graphite.
     This is the only place card text changes color.

  7. SECTION NUMBER ACCENT (current section only)
     The zero-padded section number (01, 02...) of the section
     currently in the viewport becomes var(--red-500).
     All other section numbers stay var(--muted-foreground).

  8. LOADING SCREEN CURSOR / BLINK
     The blinking cursor element on the loading screen text reveal.
     Color: var(--red-500). This is the first red the user sees.

NOT PERMITTED — red must never appear on:
  □ Backgrounds (no red surfaces, ever)
  □ Cards or panels
  □ Body copy or paragraph text
  □ Section headings
  □ Skill badges or tech stack tags
  □ Borders on cards (only on the active hover state above)
  □ Icons in their resting state
  □ Any decorative element that is not interactive

The rule: red only appears where a user is about to interact,
is currently interacting, or should be directed to interact.
If it doesn't guide action, it doesn't get red.

─────────────────────────────────────────
BUTTON SYSTEM
─────────────────────────────────────────

Three button variants. Only these three. No others.

PRIMARY — full red fill
  Use for: the single most important CTA per section (e.g. "View Work" in Hero)
  Background: var(--red-500)
  Text: oklch(0.98 0.005 260), font-mono, uppercase, tracking-widest
  Border: none
  Border-radius: 0.25rem (sharp — matches theme)
  Padding: 12px 28px
  Hover: --red-400 background + glow
  Active: --red-600 background
  Framer Motion: whileHover scale 1.02, whileTap scale 0.98
  One per section maximum. Never two red-filled buttons side by side.

GHOST — red border, transparent fill
  Use for: secondary CTAs ("Download CV", "See All Projects")
  Background: transparent
  Border: 1px solid var(--red-500)
  Text: var(--red-400), font-mono, uppercase, tracking-widest
  Hover: subtle red fill oklch(0.58 0.22 27 / 0.08) + border brightens to --red-400
  Framer Motion: whileHover scale 1.01, whileTap scale 0.98

GHOST STEEL — graphite border, no red
  Use for: low-priority actions ("View Source", "Read More")
  Background: transparent
  Border: 1px solid var(--border)
  Text: var(--muted-foreground), font-mono, uppercase, tracking-widest
  Hover: border becomes var(--graphite-400), text becomes var(--foreground)
  No red. These actions don't compete for attention.

Apply all three as CVA variants in src/components/atoms/Button.tsx.
Never create a fourth button variant. If something doesn't fit these
three, reconsider whether it needs a button at all.

─────────────────────────────────────────
TYPOGRAPHY — MONO LAYER
─────────────────────────────────────────

Two fonts only.

Display + Headings: "Geist Mono" or "JetBrains Mono"
  — monospaced, engineered, makes headings feel like terminal output
  — weight 500 (medium), letter-spacing -0.02em on large sizes
  — use for: h1, h2, navbar logo, section numbers, stat counters

Body + UI: "Geist" or "Inter"
  — clean sans, high legibility, pairs with mono headings
  — weight 400 regular, 500 for labels

Load via next/font/google:
  import { Geist, Geist_Mono } from 'next/font/google'
  const geistSans = Geist({ subsets: ['latin'], variable: '--font-sans' })
  const geistMono = Geist_Mono({ subsets: ['latin'], variable: '--font-mono' })

Apply on <html>: className={`${geistSans.variable} ${geistMono.variable}`}

tailwind.config.ts:
  fontFamily: {
    sans: ['var(--font-sans)'],
    mono: ['var(--font-mono)'],
  }

Typography rules:
  • Section headings: font-mono
  • Body copy: font-sans
  • Buttons: font-mono, uppercase, tracking-widest
  • Code, badges, stat numbers: font-mono
  • Letter spacing on display: -0.03em
  • Line height on headings: 1.1
  • No italic text — italic reads warm, this theme is cold

─────────────────────────────────────────
LAYOUT RULES
─────────────────────────────────────────

Grid-based. Everything snaps to a visible structure.

12-column grid across all sections.
Section containers: max-w-7xl, centered, px-6 (desktop) px-4 (mobile).
Sections separated by 1px horizontal rules (var(--border)), not margin alone.

Section numbering — prefix every section heading:
  <span class="font-mono text-sm text-muted-foreground">01</span>
  <h2 class="font-mono">About</h2>

The section number of the currently visible section turns red
(var(--red-500)) via an IntersectionObserver. All others stay muted.
This is a subtle but powerful signal that ties the red system together.

Whitespace: generous. Minimum 120px vertical padding per section.

─────────────────────────────────────────
SURFACE TREATMENT
─────────────────────────────────────────

Backgrounds: only --background, --card, --muted. No gradients on surfaces.
Borders: 1px solid var(--border). Cards use border, not shadow.
Border radius: 0.25rem — sharp corners everywhere except badges (pill).

Card hover state:
  border-left: 2px solid var(--red-500) — slides in from width 0 over 200ms
  background: transitions to oklch(0.14 0.009 260) over 200ms
  title text: transitions to var(--red-400) over 200ms
  No scale. No shadow. Just the red left-border and the subtle bg shift.

─────────────────────────────────────────
GLOW EFFECTS
─────────────────────────────────────────

Two glow types. Only these two.

GRAPHITE GLOW — ambient, passive:
  box-shadow: 0 0 40px -10px oklch(0.55 0.01 260 / 0.4)
  Apply to: hero 3D scene canvas only

RED GLOW — active, interactive:
  box-shadow: 0 0 30px -8px oklch(0.58 0.22 27 / 0.5)
  Apply to: primary CTA button on hover only

Nowhere else. Every additional glow dilutes both.

─────────────────────────────────────────
THINGS THAT BREAK THIS THEME (never do these)
─────────────────────────────────────────

□ Warm grays (oklch hue > 50) — reads as generic dark theme immediately
□ Any color other than red and graphite — no blue, purple, teal, green, orange
□ Red on surfaces, backgrounds, or headings — red is for interactions only
□ Two red primary buttons visible at the same time
□ Gradient backgrounds on sections
□ Card corners > 0.25rem
□ Box shadows instead of borders (except the two approved glows)
□ Personality fonts — mono + sans only
□ Colored skill badges — graphite border + muted text only
□ Emoji anywhere
□ Centered body text
□ Hover scale on cards — use red left-border transition instead
□ Red used more than 3 times in a single viewport

SELF-CHECK BEFORE ANY SECTION IS DONE
─────────────────────────────────────────

□ Desaturate to 0% — does it still look intentional? (It must)
□ Add saturation back — is red the only color that appears?
□ Are all card corners sharp (≤ 0.25rem)?
□ Are section headings in font-mono?
□ Are buttons uppercase font-mono?
□ Is the primary button red-filled?
□ Is red appearing only on interactive elements?
□ Are there any warm gray tones or any non-red colors?
□ Does the current section's number show in red?
□ Does it look engineered, not designed?
□ Would this fit in a premium developer tool?

If any answer is no — fix it before moving forward.
